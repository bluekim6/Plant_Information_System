PRD

Plant Information System
Product Requirements Document (회사 문서형)
1. 문서 개요

문서명: Plant Information System PRD
목적: 해양플랜트 설계 및 운영 단계에서 Equipment 관련 Engineering 정보를 통합 관리하고, 사용자별 검색·조회·도면 열람 기능을 제공하는 시스템의 요구사항을 정의한다.
대상 독자: 사업부서, 설계부서, IT개발팀, QA, 발주처 대응 담당자

2. 추진 배경

해양플랜트 프로젝트에서는 Equipment별로 데이터시트, 기술사양서, 설계도면, 검토의견, Vendor 문서, 변경이력 등 다수의 Engineering 정보가 생성된다.
현재 이러한 정보는 문서 저장소, 도면 폴더, 메일, 개별 관리 파일 등 여러 위치에 분산될 가능성이 높아, 설계자는 정보 추적에 많은 시간을 소요하고 발주처 및 운영자는 필요한 설비 정보를 신속하게 찾기 어렵다.

특히 Equipment 정보는 단순 문서 중심이 아니라 System → Package → Tag 의 계층 구조를 가지고 있으므로, 단순 파일 검색 방식으로는 업무 효율 확보에 한계가 있다.
따라서 본 시스템은 해당 계층 관계를 기반으로 정보 탐색이 가능하고, 사용자가 특정 System 또는 tag를 클릭했을 때 관련 설계정보 및 도면을 즉시 조회할 수 있도록 구축되어야 한다.

3. 제품 비전

Plant Information System은 해양플랜트 설계 및 운영 정보를 Equipment 중심으로 구조화하여 연결하고, 사용자가 최소한의 클릭으로 필요한 Engineering 정보와 연관 도면에 도달할 수 있도록 지원하는 통합 정보 검색 플랫폼이다.

4. 제품 목표
Equipment 관련 Engineering 정보를 통합 저장·관리한다.
System, Package, Tag 간 계층 관계를 명확히 제공하고, 하이라키 구조로 System을 클릭했을 때 System이 가지고 있는 Package가 나열되고, Package 를 클릭했을 때 Package가 가지고 있는 Tag가 List up 표현
tag 클릭 시 관련 설계정보를 즉시 불러온다.
연관된 도면을 함께 조회 및 열람할 수 있도록 한다.
도면은 'Document_Storage' 폴더에 PDF 파일로 저장되어 있고, 이것을 참조 한다.
설계자와 발주처/운영자가 동일 시스템에서 역할별로 활용할 수 있도록 한다.
Revision 및 변경 이력을 추적 가능하게 한다.

7. 핵심 개념 및 데이터 구조
7.1 계층 구조
System: 최상위
Package: Tag 상위 단위 묶음
Tag: 최하단 식별단위
7.2 관계
하나의 System은 여러 Package를 포함한다.
하나의 Package는 여러 Tag를 포함한다.
하나의 Tag는 여러 문서와 도면에 연결될 수 있다.

8. 주요 업무 시나리오
시나리오 A. 설계자

설계자가 System 과 Package 를 검색하고 해당 Tag 상세 화면으로 이동한다.
화면에서 하위 Tag, 관련 문서, 최신 Revision, 연관 도면을 확인한다.

시나리오 B. 운영자

운영자가 특정 Equipment의 Tag를 검색한다.
관련 설비 정보, 과거 검토 내용, 최신 도면을 조회하여 운영 또는 유지보수에 활용한다.

시나리오 C. 발주처

발주처가 Package 단위로 설계 정보를 탐색하고, 소속 Tag 및 관련 문서·도면을 조회한다.

시나리오 D(매우중요)
Tag를 클릭하면 화면 중앙에 Tag가 가지고 있는 정보를 표시해 주고,
그 화면에서 Tag를 한번더 클릭하면 Tag 와 연관된 도면 List를 보여 준다.
도면 No.를 클릭하면 그 해당 도면을 열람한다.
그리고 그 도면이 가지고 있는 Tag 의 리스트를 화면에 표시 해 준다.
다시 Tag를 클릭하면 Tag가 들고 있는 정보를 표시 해주는 화면으로 돌아 온다
Tag에는 Manufacture Name이 있는데, Manufacture Name 을 클릭하면 이것에 대한 상세 정보를 가지고 와서 pop up 해준다.
상세 정보는 별도 "Manufacture_list.xlsx" 에 있다

9. 핵심 요구사항
9.1 사용자 인증 및 권한
시스템 로그인 기능 제공
역할별 접근 권한 차등 제공
외부 사용자 읽기 중심 권한 부여
문서 다운로드 권한 제어
9.2 Equipment/Tag 조회
Tag No, Equipment Name, Package, 문서번호, 도면번호로 검색 가능해야 한다.
Tag 상세 화면에서 관련 문서, 관련 도면을 조회 가능해야 한다.
9.3 Comment 입력
Tag 별 Comment 를 입력 관리 할 수 있도록 한다.
9.3 Comment 조회
Comment 클릭 시 상세 내용, 작성자, 상태, 연결 문서, 연결 도면 표시
Comment 상태는 Open / Review / Closed 관리 가능해야 한다.
9.4 Package 조회
Package 상세 화면에서 소속 Tag 목록 조회
Package 단위 관련 문서 및 도면 조회

9.5 설계정보 조회
클릭한 comment 또는 tag가 보유한 설계정보를 즉시 로딩
설계정보 예시:
Tag
Description
Reference Drawing
Manufacture Name
Attribute A
Attribute B

9.6 도면 열람
관련 도면 목록 표시
PDF 중심 열람 지원
도면번호, 제목, Revision 표시
새 창 또는 내장 뷰어 제공

9.7 변경 이력
누가/언제/무엇을 변경했는지 추적
Revision 이력 조회 가능

10. 비기능 요구사항
일반 검색 응답시간 3초 이내
상세 페이지 주요 정보 2초 이내 노출
권한 기반 보안 적용
접속 및 조회 로그 저장
향후 프로젝트 확장 대응 가능 구조
다량의 도면 및 문서 연계 가능

11. 성공 기준
정보 검색 시간 단축
도면 및 문서 접근성 향상
설계 검토 및 운영 활용도 증가
중복 문의 및 정보 재탐색 감소
사용자 만족도 향상

12. 기술 방향
Frontend: React 권장
Backend: FastAPI 또는 Spring Boot
DB: DB 형태가 아닌 엑셀로 관리
